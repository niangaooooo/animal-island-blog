# 安静代码笔记

写代码时最容易丢掉的不是答案，而是抵达答案的路径。几天后再回头，只记得“最后这样能跑”，却忘了中间排除了哪些坑。

## 记录问题的形状

一篇好的技术笔记可以很短。它只需要回答三件事：

- 当时发生了什么
- 哪个判断真正改变了方向
- 下次遇到类似问题要先检查哪里

这样的笔记不追求完整教程，更像岛上的地图标记：不用解释整片森林，但要告诉你哪条路有桥。

## 保持可读而不是保持正式

个人博客最好的地方，是它不用像文档那样完美。可以写一点犹豫、尝试、误判和回头看。那些不够正式的部分，往往最能帮未来的自己恢复现场。

## 小结

如果一个问题值得你调试半小时，它通常也值得你写五分钟。下一次你会感谢这个小标记。
